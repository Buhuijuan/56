# Unity 前端文件与后端接口对照说明

本文档用于说明：

- Unity 前端里哪些文件会调用后端
- 每个文件分别调用了哪些接口
- 调用发生在什么时机
- 调用成功后会更新哪些前端状态
- 前端联调时需要特别注意什么

适用目录：

- Unity 前端脚本目录：`Assets/Scripts/UI/Scripts`
- Spring Boot 后端目录：`backend`

## 1. 总体调用链

Unity 前端当前的标准调用链大致是：

1. 某个页面或系统脚本触发操作
2. 调用 `BackendFacade`
3. `BackendFacade` 调用 `ApiClient`
4. `ApiClient` 向服务器发 HTTP 请求
5. 后端返回 JSON
6. `BackendFacade` 把返回结果写入：
   - `SessionStore`
   - `GameStateStore`
   - `BackendTaskStore`
   - 本地玩法系统
7. UI 再读取这些状态刷新显示

有一个例外：

- [AIDialoguePanelUI.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/UI/Game/AI/AIDialoguePanelUI.cs)

这个文件没有走 `BackendFacade`，而是自己直接用 `UnityWebRequest` 请求 `/api/campus/qa`。

## 2. 核心基础设施文件

这些文件不是页面，但它们决定了所有后端请求怎么发。

### 2.1 [BackendSettings.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/Backend/BackendSettings.cs)

作用：

- 保存后端开关 `UseBackendMode`
- 保存自动登录开关 `AutoLoginEnabled`
- 保存后端地址 `BaseUrl`

关键点：

- 默认地址是 `http://localhost:8080`
- 会自动把末尾 `/` 去掉
- 如果你填了 `.../api`，它会自动裁掉 `/api`

这意味着：

- 前端配置的应该是服务根地址，例如 `https://api.xxx.com`
- 不要把 `BaseUrl` 设置成 `https://api.xxx.com/api`

### 2.2 [ApiClient.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/Backend/ApiClient.cs)

作用：

- 真正发 HTTP 请求
- 支持 `GET`、`POST`、`PUT`
- 自动拼接 `BackendSettings.BaseUrl + path`
- 自动设置 `Content-Type: application/json`
- 如果有 token，自动带 `Authorization: Bearer <token>`
- 401/403 时自动执行未登录处理

调用的接口：

- 它本身不固定绑定某一个接口
- 所有标准接口最终都经过它

前端注意：

- 这里只会自动加 token，不会自动刷新 token
- 只要 401/403，就会走 `BackendRuntime.HandleUnauthorized()`，把用户踢回登录页

### 2.3 [BackendFacade.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/Backend/BackendFacade.cs)

作用：

- 是前端统一的后端接口封装层
- 页面和系统脚本基本都通过它访问后端

它封装的接口如下：

- `POST /api/auth/send-code`
- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/reset-password`
- `GET /api/player/me`
- `POST /api/player/roles`
- `PUT /api/player/current-role`
- `GET /api/game/home`
- `GET /api/tasks`
- `GET /api/tasks/current/main`
- `POST /api/tasks/{taskCode}/accept`
- `POST /api/tasks/{taskCode}/claim`
- `POST /api/tasks/events`
- `GET /api/signin`
- `GET /api/activities/clockin/current`
- `POST /api/activities/clockin/{locationId}/check`
- `GET /api/growth`
- `POST /api/growth/{stageId}/claim`
- `POST /api/growth/refresh`
- `POST /api/signin/daily`
- `POST /api/signin/online/{awardId}/claim`
- `POST /api/signin/total/{awardId}/claim`
- `POST /api/player/heartbeat`

### 2.4 [SessionStore.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/Backend/SessionStore.cs)

## 5. 目前前端需要特别注意的事情

这是最重要的一部分，建议前端同学逐条确认。

### 5.1 `BaseUrl` 必须改成服务器地址

当前默认是：

- `http://localhost:8080`

文件：

- [BackendSettings.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/Backend/BackendSettings.cs)

如果不改：

- 真机、局域网联调、正式环境都会继续请求本地

正确做法：

- 设置成类似 `https://api.your-domain.com`

不要这样填：

- `https://api.your-domain.com/api`

### 5.2 AI 对话接口是单独写死的，必须单独改

文件：

- [AIDialoguePanelUI.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/UI/Game/AI/AIDialoguePanelUI.cs)

当前问题：

- `apiUrl` 写死成了 `https://192.168.0.127:8443/api/campus/qa`

风险：

- 即使别的接口都接到新服务器了，AI 问答仍然可能访问旧地址

建议：

- 改成统一使用 `BackendSettings.BaseUrl + "/api/campus/qa"`
- 尽量也纳入 `BackendFacade`

### 5.3 登录后不要只存 token，要恢复完整会话

当前正确流程已经在做：

- 登录
- 保存 token
- `GET /api/player/me`
- 如果存在当前角色，再 `GET /api/game/home`

前端不要偷懒改成：

- 登录成功后直接进主场景

否则风险：

- 首页状态为空
- 当前角色没初始化完成
- 任务、签到、活动页显示异常

### 5.4 任务系统不要自己判定完成

任务系统正确做法：

- 前端上报事件
- 后端决定任务推进、完成、奖励

关键文件：

- [TaskEventRuntime.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/Systems/TaskEventRuntime.cs)

前端不要做的事：

- 自己直接本地标记任务已完成
- 自己本地推进主线状态

### 5.5 某些操作后必须二次刷新

有些接口不能只看一次返回值，还要刷新页面数据。

当前代码里已体现的典型场景：

- 任务事件上传后：
  - `GET /api/tasks`
  - `GET /api/tasks/current/main`
- 角色创建后：
  - `GET /api/game/home`
- 打卡成功后：
  - `GET /api/activities/clockin/current`
  - `GET /api/game/home`
- 成长奖励领取后：
  - `GET /api/growth`
- 每日签到后：
  - `GET /api/signin`

这意味着：

- 前端接新页面时，不要假设“操作接口的返回值就足够刷新整个页面”

### 5.6 401 / 403 会把用户踢回登录页

在 [ApiClient.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/Backend/ApiClient.cs) 里：

- 只要收到 401 或 403
- 就会执行 `BackendRuntime.HandleUnauthorized()`

结果：

- 清空 `SessionStore`
- 清空 `GameStateStore`
- 跳回 `01_Login`

前端要注意：

- 不要在 UI 层再重复做一遍冲突的未登录处理

### 5.7 `UseBackendMode` 会决定是否真正走服务器

文件：

- [BackendSettings.cs](/C:/Users/10034/Desktop/3.17/MyCampus1/Assets/Scripts/UI/Scripts/Backend/BackendSettings.cs)

如果它是 `false`：

- 很多页面会走本地假数据或本地系统
- 你以为自己在联调服务器，其实没有请求后端

联调前必须确认：

- `UseBackendMode == true`

### 5.8 某些后端接口虽然有，但前端暂时没接

比如：

- 修改邮箱
- 修改密码
- 切角色的完整 UI
- 资产模块接口
- 问答活动完整接口
- 故事活动完整接口

如果后续要接：

- 优先在 `BackendFacade` 中增加统一封装
- 不要在页面里散落直接写 `UnityWebRequest`

### 5.9 HTTPS 证书处理要谨慎

当前 AI 文件里用了：

- `BypassCertificate`

这在正式环境有风险，因为它会忽略证书校验。

正式部署建议：

- 用有效 HTTPS 证书
- 去掉绕过证书逻辑

### 5.10 DTO 与后端字段名必须保持一致

后端 JSON 字段是按当前 DTO 输出的。

前端如果自己新增 DTO，需要特别注意：

- 字段名大小写
- 可空字段
- 某些接口返回 `data == null` 的情况

尤其要注意两个不同返回体系：

- 大多数接口：`{ success, data, message }`
- AI 问答接口：`{ code, msg, request_id, player_id, data }`
